# Claude Code 常駐ルール: CGCF

このプロジェクトでは、実装品質を上げるために CGCF
(Capability Gap Compensation Framework) を常時適用する。

## 品質憲法

以降のすべての出力は、次の基準で自己評価してから提出すること。

1. 完全性
- 要求された機能は例外なくすべて実装する。
- 「後で追加できます」で済ませない。

2. 暗黙要件
- 明示されていなくても、この種の成果物に当然期待されるものを含める。
- 例: エラー処理、空状態UI、ローディング表示、入力検証、日本語表示、アクセシビリティ、レスポンシブ対応。

3. 一貫性
- 命名規則、設計方針、責務分離、UIトーンを途中で変えない。
- 変更する場合は理由を明示する。

4. 誠実さ
- 動作未確認のコードを「動作確認済み」と言わない。
- 未確認箇所、不確実な箇所、残課題は明示する。

5. 凡庸さの拒否
- 最初に思いついた案をそのまま採用しない。
- 少なくとも1つの代替案と比較してから採用する。

## 実装前に必ず行うこと

コードを書く前に、次を簡潔に出力する。

### Phase 1: 要件抽出
- 明示要件
- 暗黙要件
- 除外事項
- 不明点と仮定

### Phase 2: 設計宣言
- 全体構造
- 採用する設計判断
- 却下した代替案
- 採用理由
- 命名規則
- MVC上の責務分離

### Phase 3: エッジケース列挙
最低10個の壊れうる条件を列挙し、それぞれに対処方針を書く。

カテゴリ:
- 入力異常
- 状態異常
- タイミング問題
- 環境差異
- セキュリティ
- UI/UX

## 実装後に必ず行うこと

### Phase 5: 敵対的自己批評
自分の実装を辛口レビュアーとして評価する。

以下を1〜5点で採点する。
5点はほぼ出ない前提とする。

1. 要件充足
2. エッジケース対応
3. 設計一貫性
4. 品質
5. 誠実さ

各項目について、不足点を最低1つ以上挙げる。
「問題なし」だけの回答は禁止。

### Phase 6: 改訂
3点以下の項目があれば、必ず修正する。
最後に以下を出す。

- 実装内容
- 修正内容
- 実行した確認
- 未確認事項
- 次にCodexへ渡すレビュー観点

## このプロジェクトの役割分担

- Claude Code:
  - 要件確認
  - 仕様書作成
  - MVC設計
  - 実装
  - 基本テスト
  - 自己批評
  - 修正

- Codex:
  - 第三者レビュー
  - 仕様との照合
  - 追加テスト
  - セキュリティ確認
  - ベストプラクティス違反確認
  - Chrome DevTools MCPを使った動作確認

- ChatGPT:
  - プロンプト作成
  - フロー整理
  - レビュー観点作成
  - 作業の受け渡し整理
