# floating-islands-rpg

## 概要

浮遊島を舞台にした短編ターン制コマンドRPGです。

村・フィールド・ダンジョンという古典的な3エリア構成の「浮遊島」を舞台に、探索→戦闘→成長→物語進行というRPGの基本ループを、短時間(1周30〜60分想定)で過不足なく体験できることを目指すプロジェクトです。

**現在はMVP(Minimum Viable Product)の中核ループが一通り実装済みの段階です。詳細は次節「現在の完成範囲」および「既知の制限」を参照してください。**

## 遊び方

### 起動・セーブ・ロード

| 操作 | 説明 |
|---|---|
| New Game(はじめから) | タイトル画面から新規にゲームを開始する。Villageから始まる。 |
| Continue(つづきから) | タイトル画面から直前のセーブデータを読み込んで再開する。セーブデータが存在しない、または壊れている場合はエラーを通知したうえでタイトルに留まる。 |
| Save(セーブ) | セーブ処理自体(`SaveGameUseCase`)はT-010/T-011で実装済みだが、**プレイヤーが任意のタイミングで手動保存するためのゲーム内UI(セーブボタン等)は現時点で未実装**(下記「既知の制限」参照)。 |

### 移動・会話

- 移動: `W`/`A`/`S`/`D` または矢印キー(ゲームパッド左スティックにも対応)。
- NPCとの会話: NPCに近づいて `E` キー(会話ウィンドウが開き、入力で読み進める)。

### Menu(メニュー)

Village/Field/Dungeonのいずれからも、画面上の「Menu」ボタンで開き、「Close」ボタンで閉じる共通メニューを呼び出せる。メニュー表示中はプレイヤー移動・エンカウント判定・NPC会話・Scene遷移が一時停止し、閉じると復帰する。

- **Inventory(アイテム)**: 所持しているPotionの一覧・所持数・効果を確認し、「Use」ボタンでその場でHPを回復できる。
- **Equipment(装備)**: 所持しているWeapon/Armorの一覧を確認し、「Equip」で装備、装備中の項目は「Unequip」で解除できる。装備するとステータス(攻撃力・防御力)に反映される。

### Quest(メインクエスト)/ SubQuest(サブクエスト)

- **メインクエスト**: Villageの指定NPCと会話すると開始する。以後、フィールド到達・ダンジョン到達・ボス撃破のたびに自動的に進行し、ボス撃破時にメインクエストが`DefeatBoss`段階に達していれば完了しゲームクリアとなる。
- **サブクエスト(2本)**: Village内の別々のNPC(会話専用)からそれぞれ独立に受注できる。メインクエストの進行状況に一切関係なく受注・進行・完了でき、サブクエスト1はフィールド到達、サブクエスト2はダンジョン到達で自動的に完了する。

### Battle(戦闘)

フィールド・ダンジョンの探索中にランダムエンカウントで発生するほか、ダンジョン最奥のボス部屋に入ると必ずボス戦が発生する。戦闘はターン制で、行動順は素早さ等のステータスで決まる。**現時点ではコマンドは「たたかう」のみ実装済み**(とくぎ/どうぐ/にげるはUI未実装、下記「既知の制限」参照)。勝利すると経験値とアイテム報酬(初回設定分)を獲得し、条件を満たすとレベルアップする。

### Retry(敗北時の再挑戦)

戦闘に敗北するとゲームオーバー画面へ遷移し、「Retry」で同じ戦闘(通常戦/ボス戦の別、および復帰先のフィールド/ダンジョン)を、戦闘直前のプレイヤー状態(HP等)から再挑戦できる。

### GameClear条件

メインクエストが`DefeatBoss`段階まで進んだ状態でダンジョンのボスを撃破すると、メインクエストが`Completed`となりゲームクリア画面へ遷移する。ボスを撃破してもメインクエストが`DefeatBoss`段階に達していない場合(メインクエスト未着手のままボス部屋に到達した場合等)はゲームクリアにはならず、フィールド/ダンジョンへ復帰する。

## 既知の制限

- ゲーム内にセーブを実行するためのUI(セーブボタン等)が存在しない。`SaveGameUseCase`自体は実装・テスト済みだが、プレイヤーが任意のタイミングで手動保存する導線は未実装。
- 戦闘コマンドは「たたかう」のみ実装済み。「とくぎ/どうぐ/にげる」はPROJECT.mdの想定仕様に含まれるが、UI上は未実装。
- 「つづきから(Continue)」は保存時点のシーンに関わらず、常にVillageへ復元される(`PlayerSessionState.MoveToScene`が現状どのインストーラーからも呼び出されていないため)。
- 通貨・ショップ・売買・クラフト・装備耐久度・ランダムオプション・アイテムソート等は未実装(T-026で意図的にスコープ外とした範囲)。
- モデル・テクスチャ・音源・ボイス等の演出面はUnity標準機能とプレースホルダー中心で、独自アートワーク・音声収録は行っていない。
- 想定プレイ時間(30〜60分)は自動テストによる通し確認で完走できることを確認しているが、ストップウォッチ等による人手の時間計測は実施していない。
- v0.1.0のWindowsビルド(T-029/T-030)は、実行ファイルの起動・Title画面描画・New Gameクリックによるシーン遷移を実機で確認済みだが、Menu/Save/Continue/Battle/Retry/Dungeon/Boss戦/GameClear/再起動後Continueをビルド済み実行ファイル上でのインタラクティブな手動操作では確認できていない(自動化されたPlayModeテスト[実SceneをロードするE2Eテスト含む、3回連続Passed]による代替検証のみ)。詳細は [docs/KNOWN_LIMITATIONS.md](./docs/KNOWN_LIMITATIONS.md) を参照。

## 開発環境

- Unity Hub + Unity 6000.3.17f1
- URP (Universal Render Pipeline)
- C# / 新Input System
- Unity MCP(`com.coplaydev.unity-mcp`、Claude Code等のAIエージェントからUnity Editorを操作するための接続基盤。実装セッションによっては未接続の場合もあり、その場合はUnity Editorのコマンドラインバッチモードでテストを実行している)
- OS: Windows
- IDE: Visual Studio Code / Visual Studio / Rider いずれか(`.csproj` / `.sln` はUnityが自動生成するためGit管理対象外)

### Unity Hubから開く方法

1. Unity Hubを起動する。
2. 「開く」からこのリポジトリのルートディレクトリ(`floating-islands-rpg/`、`Assets/`や`ProjectSettings/`があるフォルダ)を選択する。
3. Unity 6000.3.17f1 がインストールされていない場合はUnity Hub経由でインストールする。
4. プロジェクトを開くとパッケージの解決・インポートが自動的に行われる。

### テストの実行方法

- Unity Editorを開いた状態で `Window > General > Test Runner` からEditMode/PlayModeそれぞれのテストを実行する。
- または、Editorを閉じた状態でコマンドラインから以下のように実行する(`-runTests` 使用時は `-quit` を併用しないこと。同時指定するとスクリプトコンパイル完了直後にEditorが終了しテストが実行されない)。

```
Unity.exe -batchmode -nographics -projectPath <このリポジトリのパス> -runTests -testPlatform EditMode -testResults <出力先>.xml -logFile <ログ出力先>.log
Unity.exe -batchmode -nographics -projectPath <このリポジトリのパス> -runTests -testPlatform PlayMode -testResults <出力先>.xml -logFile <ログ出力先>.log
```

## 開発フロー

本プロジェクトは以下の開発フローに従います。

1. **ImplementerとReviewerを別コンテキストにする**: 実装を行うエージェント/担当と、レビューを行うエージェント/担当のコンテキストを分離し、実装者の思い込みをそのまま通さない。
2. **PROJECT.md承認後に実装**: 設計・仕様・要承認事項が人間によって承認された後に、対応するタスクの実装へ着手する。
3. **featureブランチ**: `main`へ直接コミットせず、タスク単位でfeatureブランチを作成して作業する。
4. **テスト**: Domain/Applicationのロジックは EditMode テスト、シーン遷移や戦闘フロー等の結合的な動作は PlayMode テストで検証する。
5. **第三者レビュー**: 実装後、Implementerとは別のレビュープロセスでコードレビューを行う。
6. **PR**: レビューを経てPull Requestを作成し、`main`へマージする。
